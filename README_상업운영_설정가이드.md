# M.O.T. 상업운영 구현 가이드

이 폴더는 기존 `mot-printcore.com` GitHub Pages 홈페이지에 **계약·견적·결제·세금정산·PDF/영상 판매** 구조를 추가한 배포본입니다.

## 1. 이번에 추가된 기능

| 파일 | 역할 |
|---|---|
| `commerce.html` | B2B 서비스 가격표, 계약·결제 흐름, PDF·영상 상품 구조, 견적요청 페이지 |
| `terms.html` | 이용약관·기술서비스 조건 템플릿 |
| `refund.html` | 디지털 콘텐츠 환불·구독정책 템플릿 |
| `privacy.html` | 개인정보 처리방침 템플릿 |
| `assets/js/commerce-config.js` | 사업자 정보, 공개 이메일, 외부 결제 URL을 한 번에 설정하는 파일 |
| `assets/js/commerce.js` | 가격 카드, 결제 버튼, 견적요청 메일 자동작성 기능 |
| `assets/css/commerce.css` | 상업운영 페이지 전용 디자인 |
| `assets/downloads/MOT_기술서비스_기본계약서_초안.docx` | 편집 가능한 B2B 기술서비스 계약서 |
| `assets/downloads/MOT_기술서비스_견적의뢰서_프로젝트브리프.docx` | 고객에게 보내는 견적의뢰서 |
| `assets/downloads/MOT_세무정산_월간관리대장.xlsx` | 매출·플랫폼수수료·비용·증빙을 월별로 정리하는 관리대장 |

## 2. 가장 먼저 해야 할 5가지

1. **개인사업자 또는 법인 사업자 정보 확정**
   - 상호, 대표자, 사업자등록번호, 사업장 주소, 고객지원 이메일
   - B2C 디지털 콘텐츠를 직접 판매하는 경우 통신판매업 관련 표시·신고 요건을 확인

2. **사업용 계좌·카드 분리**
   - B2B 고객의 입금은 견적서/청구서에만 계좌를 표시
   - 개인 생활계좌와 분리하여 매출·비용·플랫폼 정산을 관리

3. **세무대리인 선정 후 판매 구조 검토**
   - 국내 B2B, 국내 B2C, 해외 B2B, 해외 플랫폼 정산을 각각 구분
   - 부가세, 전자세금계산서, 해외 용역·외화 입금·원천징수 가능성은 거래 시작 전에 확인

4. **외부 결제·콘텐츠 플랫폼 계정 개설**
   - B2B: 견적·계약·청구서·은행이체 또는 공급자 인보이스
   - PDF: 결제 완료 후 다운로드·영수증·구매자 이메일을 제공하는 디지털 판매 플랫폼
   - 영상 구독: 로그인·결제상태·권한관리·해지를 지원하는 코스/멤버십 플랫폼

5. **`assets/js/commerce-config.js`에 실제 정보 입력**
   - `{{ ... }}` 형식의 모든 항목을 실제 정보로 교체
   - `checkouts.*`에는 각 상품의 외부 결제 URL만 입력
   - 비밀키, API 키, 계좌 로그인 정보는 절대 GitHub에 입력하지 않음

## 3. 권장 판매 구조

### A. B2B 기술서비스: 계약·청구서 방식

`문의 → 필요 시 NDA → 유료 기술진단 견적 → 계약 → 청구 → 입금 확인 → 킥오프 → 산출물·검수 → 잔금`

- 기술진단: 계약 후 100% 선결제
- 8주 개선 Sprint: 착수 50% / 중간 30% / 종료 20%
- 공동개발: 착수 40% / 설계동결 30% / 시제품검증 20% / 이관 10%
- 월 기술고문: 매월 선결제
- 기업교육: 교육일 7영업일 전 100% 결제

**공개 웹사이트에 은행계좌번호를 노출하지 마십시오.** 계약 또는 확정 청구서에만 표시합니다.

### B. PDF 기술자료: 단권 다운로드 판매

- 권장: 단권 1회 결제 + 구매자 이메일 또는 주문번호 워터마크 PDF
- 상품 페이지: 미리보기 10~15페이지, 목차, 대상 독자, 버전일, 파일 형식, 환불정책 표시
- 제공: 외부 판매 플랫폼이 결제·영수증·다운로드 링크를 처리
- 가격 예시: `₩79,000 / 권` (부가세 표시방식은 사업자·플랫폼 구조에 맞춤)

### C. PDF 라이브러리: 다운로드가 아닌 열람권 구독

- 월 구독으로 전체 PDF를 무제한 다운로드하게 하면 첫 달 결제 후 해지·공유 위험이 큼
- 권장: 로그인 기반 열람권 + 월간 업데이트 문서 + 체크리스트
- 가격 예시: `₩59,000 / 월`

### D. 교육 동영상: 계정 기반 스트리밍·멤버십

- 비공개 영상 URL만 이메일로 보내지 말 것
- 반드시 로그인 계정, 결제상태, 해지, 권한만료, 시청기록을 관리하는 플랫폼 사용
- 단과 과정: `₩149,000 / 과정`
- 멤버십: `₩129,000 / 월`
- 기업 라이선스는 인원·기간·다운로드권한·녹화금지 조건을 별도 견적

## 4. 결제 플랫폼 연결 원칙

GitHub Pages는 정적 웹사이트입니다. 카드 결제·정기결제·웹훅·회원 권한을 직접 처리하지 않습니다.

따라서 홈페이지는 **브랜드·상품·설명·견적요청·결제 버튼**을 담당하고,
결제와 계정 권한은 **외부 호스티드 결제 또는 멤버십 플랫폼**이 담당합니다.

### 구성 예시

| 거래 | 홈페이지 버튼의 목적 | 실제 처리 위치 |
|---|---|---|
| 국내 B2B 기술서비스 | 견적 요청 | 이메일·NDA·계약서·청구서·계좌이체 |
| 해외 B2B 기술서비스 | 견적 요청 | 영문 견적·인보이스·해외송금 또는 인보이스 결제 |
| PDF 단권 | 외부 결제 페이지 | 디지털 판매 플랫폼의 결제·다운로드·영수증 |
| 문서 라이브러리 | 구독 결제 페이지 | 멤버십 플랫폼의 로그인·권한관리 |
| 영상 과정/멤버십 | 수강 또는 구독 결제 | 코스 플랫폼의 계정·스트리밍·구독관리 |

## 5. `commerce-config.js` 실제 입력 예시

```javascript
company: {
  legalName: "M.O.T. Technology Innovation Lab",
  representative: "홍길동",
  businessRegistrationNo: "123-45-67890",
  mailOrderRegistrationNo: "2026-서울강남-0000",
  address: "대한민국 ...",
  customerServiceEmail: "support@yourdomain.com",
  billingEmail: "billing@yourdomain.com",
  privacyOfficer: "홍길동",
  privacyEmail: "privacy@yourdomain.com"
},
checkouts: {
  pdfSingle: "https://YOUR_CHECKOUT_PROVIDER/PDF-SINGLE",
  pdfLibraryMonthly: "https://YOUR_MEMBERSHIP_PROVIDER/PDF-LIBRARY",
  academyCourse: "https://YOUR_COURSE_PROVIDER/COURSE-01",
  academyMembership: "https://YOUR_MEMBERSHIP_PROVIDER/ACADEMY"
}
```

입력 후 `{{ ... }}`가 남아 있으면 법적 고지 페이지가 미완성입니다. 반드시 모두 교체한 뒤 공개하십시오.

## 6. 월말 세금·정산 루틴

`assets/downloads/MOT_세무정산_월간관리대장.xlsx` 사용 순서:

1. `Sales Ledger`에 B2B 청구·입금, PDF·영상·구독 매출을 모두 기록
2. `Expense Ledger`에 플랫폼 수수료, 환불, 출장, 외부 시험, 사업용 카드 지출 기록
3. 세금계산서·주문번호·정산명세·인보이스·입금증빙 번호를 각 거래행에 입력
4. PDF 증빙파일을 `YYYY-MM` 폴더별로 보관
5. `Monthly Summary`와 `Dashboard`에서 매출·VAT·수수료·비용·미수를 점검
6. 세무대리인에게 원장, 정산명세, 계약서, 해외 인보이스·입금증빙을 전달

## 7. 공개 전 점검표

- [ ] 사업자 정보, 고객지원 이메일, 개인정보 문의 이메일 입력
- [ ] 약관, 환불정책, 개인정보 처리방침의 시행일 입력
- [ ] 통신판매업 관련 표시·신고 요건 검토
- [ ] B2B 계약서의 준거법·관할·IP·검수·지급조건 검토
- [ ] 결제 링크 4개를 실제 외부 체크아웃 URL로 교체
- [ ] 결제 성공 후 PDF 전달·영상 권한부여가 실제로 동작하는지 테스트
- [ ] 환불·해지·고객지원 메일 테스트
- [ ] 실제 카드번호·API 비밀키·은행 로그인 정보가 GitHub에 없는지 재확인
- [ ] 세무대리인과 국내/해외 거래의 세금 처리 확인

## 8. GitHub 업로드

이 폴더 전체를 기존 `mot-printcore.com` 저장소의 루트에 덮어씁니다.

권장 방법은 **GitHub Desktop**입니다. 폴더 구조가 유지되며 `assets/css`, `assets/js`, `assets/downloads`가 정상적으로 반영됩니다.

웹 브라우저 업로드만 사용할 경우에는 `assets` 폴더를 먼저 만든 뒤, 하위 폴더별로 파일을 업로드해야 합니다. 폴더가 누락되면 `commerce.html`의 디자인·버튼·다운로드 링크가 작동하지 않습니다.


## 9. 별도 상업·정책 페이지 다국어 운영

`commerce.html`, `terms.html`, `privacy.html`, `refund.html`은 모두 한국어·영어·중국어(간체) 전환을 지원합니다.

- 각 페이지 오른쪽 상단의 `KO / EN / 中文` 버튼으로 언어를 바꿉니다.
- 메인 홈페이지에서 선택한 언어는 브라우저에 저장되어 상업운영·약관·환불·개인정보 페이지에도 자동으로 이어집니다.
- 직접 공유할 때는 아래처럼 주소 끝에 언어를 지정할 수 있습니다.

```text
commerce.html?lang=en
commerce.html?lang=zh
terms.html?lang=en
privacy.html?lang=zh
refund.html?lang=en
```

다국어 문구는 `assets/js/commerce-i18n.js`에서 관리합니다. 상품명·서비스 범위·가격 단위·환불 문구를 수정할 때에는 `assets/js/commerce-config.js`와 함께 해당 언어의 `commerce-i18n.js` 문구도 동일하게 검토하십시오. 특히 약관·개인정보·환불 정책의 영문·중문은 실제 운영조건 및 결제 플랫폼에 맞춰 법률·세무 검토 후 확정해야 합니다.

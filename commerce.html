상업 운영 체계

기술은 무료 조언으로 소모하지 않고,
계약·성과·지식자산으로 운영합니다.

이 페이지는 M.O.T.의 B2B 기술서비스와 PrintCore Academy 디지털 콘텐츠를 실제 거래 가능한 구조로 운영하기 위한 가격, 결제, 세금 증빙, 이용권 및 계약 기준입니다.

B2B: 견적 → 계약 → 청구 → 납품 B2C: 호스티드 결제 → 계정 권한 → 영수증 GitHub Pages에는 카드정보를 저장하지 않습니다.

운영 원칙

기술서비스와 디지털 콘텐츠는
결제 구조를 분리해야 안전합니다.

컨설팅·공동개발은 고객별 범위와 책임이 달라 계약·청구서·계좌이체 중심으로 운영합니다. 반면 PDF·영상은 외부 호스팅 결제 페이지와 계정 권한관리로 운영하여, GitHub Pages에 카드정보나 비밀키를 저장하지 않습니다.

01

범위 먼저, 가격 다음

문제 정의, 대상 부품, 목표, 데이터, 현장 접근 권한을 확인한 뒤 서면 견적을 발행합니다. “성과 보장”이 아닌 검증 가능한 산출물을 계약서에 명확히 씁니다.

02

결제는 외부 결제화면에서

카드·간편결제·정기구독은 인증된 호스티드 결제 화면으로 보냅니다. 정적 GitHub Pages에는 PG 비밀키, 고객 카드정보, 비밀번호를 넣지 않습니다.

03

세금·증빙은 거래별로

국내 B2B는 견적서·계약서·전자세금계산서 또는 적법한 증빙을 연결합니다. 해외 고객·플랫폼 매출은 수수료와 정산명세를 원화 기준 장부에 함께 보관합니다.

B2B 기술서비스

초기 공개 가격 가이드라인
— 범위와 리스크에 따라 서면 견적

아래 금액은 M.O.T.의 독립 기술전문가 사업을 위한 권장 시작가격입니다. 국내 거래는 부가세가 별도 적용될 수 있으며, 출장·시료·외부 시험·통번역·운송·장비·특허 비용은 기본 가격에 포함하지 않습니다.

계약 및 결제 흐름

B2B 프로젝트의 표준 흐름

01

사전 적합성 검토

제품·문제·목표·일정·데이터를 확인합니다. 기밀자료가 필요하면 NDA를 먼저 체결합니다.

02

견적서·계약서

범위, 산출물, 고객 협조사항, 일정, 비용, 지식재산권, 비밀유지, 검수기준을 서면 확정합니다.

03

청구·입금 확인

계약에 정한 선금 또는 월 자문료를 청구합니다. 입금 확인 후 킥오프와 자료수령을 시작합니다.

04

산출물·검수·이관

보고서·DOE·표준서·교육자료를 제출하고, 정해진 검수기간 후 잔금을 청구합니다.

중요

공동개발은 기존 기술(Background IP), 고객 제공자료, 신규 개발결과(Foreground IP)의 소유·사용권을 계약서 별표로 반드시 구분하십시오. 고객 맞춤 최종 산출물의 사용권은 대금 완납 후 부여하는 구조를 권장합니다.

PrintCore Academy

지식자산 판매 구조
— 다운로드와 구독권을 구분합니다.

다운로드된 PDF는 회수가 사실상 불가능하므로 단권 판매로 운영합니다. 월 구독은 로그인 기반 열람권·스트리밍 권한으로 운영해야 해지 후 권한을 관리할 수 있습니다. 실제 결제 링크는 아래 설정 파일에 입력하면 버튼에 연결됩니다.

결제 및 세금 운영

GitHub Pages에서 가능한
현실적인 결제·정산 구조

01

사업자·전용계좌·장부 먼저 정리

사업자등록, 사업용 계좌·카드, 세금계산서 발행 체계, 월간 매출·수수료·비용 증빙 폴더를 먼저 만듭니다. 국내 B2B 계약은 견적서 → 계약서 → 청구서/전자세금계산서 → 입금확인 순서로 관리합니다.

02

B2B는 “결제버튼”보다 청구서

고객별 과제는 표준 가격표를 기준으로 견적을 확정한 뒤, 은행이체 또는 공급자 대시보드의 개별 청구서 링크로 수금합니다. 계좌번호는 공개 웹사이트가 아니라 확정 청구서에만 표기하십시오.

03

PDF는 구매·전달 플랫폼에 맡기기

결제 완료 후 개별 다운로드 링크, 영수증, 구매자 이메일을 처리하는 디지털 판매 플랫폼을 사용하십시오. 파일에는 구매자 이메일 또는 주문번호 워터마크를 넣어 무단공유를 억제합니다.

04

영상 구독은 로그인·권한관리 필수

비공개 영상 URL만 전달하면 공유를 막을 수 없습니다. 회원 계정, 결제상태, 구독해지, 시청 권한을 관리하는 코스·멤버십 플랫폼을 연결하고, GitHub Pages는 소개·유입·결제 버튼 역할만 맡깁니다.

05

국내·해외 매출을 분리 기록

국내 매출은 세금계산서·현금영수증·PG 정산자료를, 해외 고객·해외 플랫폼 매출은 인보이스·정산명세·환전 또는 외화입금 증빙을 거래별로 보관합니다. 해외 용역의 세무처리는 계약조건에 따라 달라질 수 있으므로 발행 전 세무전문가에게 확인하십시오.

사이트 연결 위치

실제 결제 URL, 사업자 표시사항, 고객지원 이메일은 한 파일에서 관리합니다. 값 입력 후 GitHub에 다시 업로드하면 버튼과 법적 고지 페이지에 반영됩니다.

assets/js/commerce-config.js company.businessRegistrationNo company.mailOrderRegistrationNo company.customerServiceEmail checkouts.pdfSingle checkouts.pdfLibraryMonthly checkouts.academyCourse checkouts.academyMembership

결제 서비스의 비밀키·웹훅 키·API 키는 이 파일에 절대 넣지 마십시오. GitHub 저장소는 공개될 수 있습니다.
